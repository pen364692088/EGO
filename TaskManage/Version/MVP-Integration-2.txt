Integration-2 总目标

实现一个可控、可审计、可回放的真实对话闭环：

每轮必读 emotiond guidance（不依赖“模型自觉去读文件”）

关键设计决策（必须遵守）
D1. target_id 不可伪造：永远来自 hook 上下文

target_id = conversationId（Telegram）

兼容 fallback：若某些 channel 出现 ctx.conversationId 缺失（已有人在部分场景报告过），fallback 到 ctx.sessionKey 或 ${channel}:${from}，但必须写 audit/log 说明采用了 fallback

D2. “每轮必读”靠 OpenClaw 的 injected workspace files 实现

OpenClaw 默认会注入固定文件列表（TOOLS.md / USER.md / IDENTITY.md / ...）
因此：不要依赖 agent 去 cat 额外文件，而是让 hook 把“本轮 emotiond runtime context”写进某个注入文件的一个小区块中（覆盖写、短文本、不膨胀）。

推荐文件：TOOLS.md（最不容易污染人格/长期记忆），用 marker 区块覆盖写：

TOOLS.md 保留你原内容

hook 每轮只覆盖：

<!-- EMOTIOND_RUNTIME_BEGIN -->
{...本轮 runtime json...}
<!-- EMOTIOND_RUNTIME_END -->

这样能保证“每轮必读”（因为文件在上下文里），同时避免长期文件无限增长。

D3. pre-turn 只做确定性工作；语义分类交给 agent

hook：time_passed、target_id、request_id_base、把 runtime context 写进注入文件

agent：分类 subtype、发送 world_event、再拉取 /decision（post-event）生成回复

B) 可直接粘进 SKILL.md 的分类契约（Classifier Contract）

把下面这一段作为 SKILL.md 的一个章节（比如 ## World Event Classification (Contract)）。

1) 输出必须是“纯 JSON”，不得带额外文本

你每次分类必须输出 完全符合 schema 的 JSON（上面那份）。

2) subtype 定义（务必按定义，而不是按情绪想象）

care：对方表达关心/支持/安慰/愿意陪伴/积极投入关系（行动或明确意图）

apology：对方承认过错/表达歉意/愿意弥补（不等于“解释原因”）

ignored：对方持续不回应/敷衍/多次跳过关键点/明显冷处理（一次没回不一定）

rejection：对方明确拒绝你/否定关系/表示不想继续/让你走开（明确边界）

betrayal（高影响）：对方违背承诺、欺骗、背刺、泄密、明显利用你（必须有强证据）

neutral：信息性沟通/日常闲聊/无法映射上述

uncertain：你不确定 subtype（建议走门控回退）

3) 高影响定义（is_high_impact）

betrayal 永远 is_high_impact = true

rejection 视强度：

明确分手/拉黑/“别再联系”→ high

礼貌拒绝/暂时不行 → non-high（更像 neutral 或 mild rejection）

ignored 通常 non-high（除非明确长期冷暴力且证据强）

4) 置信度标定（关键：别乱给 0.9）

用一个简单可回归的标定法（你可以照着打分）：

起始值：0.55
加分项（每条 +0.10，最多加到 0.95）：

有明确关键词（“对不起/抱歉/我错了”）或明确动作（“我会补偿/我来修复/我现在做X”）

有明确关系意图（“我在乎你/我会陪你/我支持你”）

有明确拒绝句式（“不要/别再/不想/结束/离开”）

有明确证据的背叛（承诺+违背/欺骗+证据）

减分项（每条 -0.10，最低到 0.05）：

明显玩笑/阴阳怪气/夸张修辞（可能是调侃）

语境不足（单句短文本、没有前后）

多义（“行吧/随你/呵呵”这种）

5) 门控规则（should_send）

betrayal：confidence ≥ 0.90 才能发送；否则 should_send=false，fallback 选 ask_clarifying_question

rejection：confidence ≥ 0.75 才发送；否则回退

ignored：confidence ≥ 0.60 才发送；否则回退到 no_event_reply_observe

care/apology：confidence ≥ 0.55 可发送

neutral/uncertain：默认不发送（除非你用 send_neutral_only 走轻量事件；但建议先别发，避免稀释信号）

6) 回退策略（fallback）

ask_clarifying_question：一句话澄清（不要质问），例如“你是想表达你需要空间，还是不想继续？”

no_event_reply_observe：不发送 world_event，只基于 emotiond 的当前 decision 走“观察/轻问/不升级冲突”

send_neutral_only：仅用于你想记录对话节奏但不想改变关系状态（可选，默认不用）

C) 常见误判对照表（强烈建议写进 SKILL.md）

“我现在很忙” ≠ ignored

若只是一次说明忙：多半 neutral

若多次无回应 + 反复跳过关键诉求：才可能 ignored

“你别闹了（笑）” ≠ rejection

调侃/缓和：可能 neutral 或低置信度 uncertain

“算了” 很多时候不是 rejection

可能是失望/疲惫（更像 ignored 的前兆），除非明确说“别联系了”

“解释原因” ≠ apology

apology 必须含承担责任/歉意/弥补意图

“我不想现在谈” 不一定是 rejection

更像 neutral 或低强度 rejection（但通常低置信度，优先澄清）
