DELIVERABLES
D1: Eval Suite v2 → v2.1 (Attribution + Telemetry)
要做什么

增强 scripts/eval_suite_v2.py（或新增 eval_suite_v2_1.py，但推荐原地增强并保留兼容）：

逐场景结果结构化
输出每个 scenario 的：

pass/fail

failure_reasons（可多标签）：

false_high_impact（误触发高影响事件）

missed_clarify（应澄清却未澄清）

over_clarify（过度澄清/反思）

ledger_misfire（承诺证据未生效/误判）

state_leak（跨 target 状态污染）

precision_saturation（权重被阈值夹死几乎不变）

budget_collapse（能量预算过快坍塌/恢复过慢）

intrinsic_dead（expected_info_gain 变化不影响策略）

key_metrics（每场景，不要只全局汇总）

中间量 Telemetry（必须）
对每轮/每 step 记录并在报告里汇总统计（均值/分位数/最大/最小/变化率）：

precision：w_external/w_internal/w_memory/w_action/w_explore

allostasis：energy_budget 曲线

intrinsic：expected_info_gain、boredom/curiosity/confusion 强度

self_model：更新次数、values 变化量、identity_stability

ledger：证据强度分布（promise_confidence、violation_strength）

decision：action 分布（approach/repair/boundary/withdraw/attack），meta-cog intents 触发频次

敏感性检查（强制）
新增一个 “parameter sensitivity smoke test”：

给定两组明显不同的参数（例如探索权重翻倍、precision 温度大幅变化）

断言 telemetry 或行为分布至少在 1~2 个维度上发生可观测变化
否则直接 fail（说明指标/记录没打到路径上）

验收

新增至少 25 个测试覆盖：失败归因、telemetry 字段完整性、敏感性 smoke test 可稳定通过

生成报告包含 per-scenario 章节 + telemetry 汇总表

D2: AutoTune v0 → v0.1 (Real Search)
要做什么

增强 scripts/auto_tune_v0.py（或新增 auto_tune_v0_1.py）：

搜索策略：两阶段

Stage A（全局粗扫）：

Random search + 分层采样（Latin Hypercube 或 Sobol 任一即可）

200–500 组候选（默认 200，支持 CLI 参数）

Stage B（局部精调）：

坐标下降 / 局部随机爬山（实现简单、可控）

在 top-k 候选附近继续 50–150 次迭代

目标函数（fitness）必须多目标加权
最小包含：

场景通过数（权重最高）

高影响误判率（越低越好）

cross_target_interference（越低越好）

过度澄清率（控制上限）

平均 token/轮（可选，控制成本）

输出

reports/auto_tune_<timestamp>.json：包含 top-N 候选列表（每个候选都带参数 + per-scenario metrics）

reports/auto_tune_<timestamp>.md：人类可读总结（最佳候选为什么好、牺牲了什么）

best_params_<timestamp>.json：可直接加载的参数文件

可复现

所有搜索都支持 --seed

报告写入：代码版本（git commit hash）、参数空间定义、场景版本号

验收

至少 10 个测试：搜索 reproducible、输出 schema 完整、错误处理清晰

至少在本地跑出 1 次“指标有变化”的候选（不要求变好，但必须能变动；否则说明闭环没打通）

D3: Cross-target Interference — Diagnose & Gate
要做什么

把 cross_target_interference 拆成可定位子项
例如：

state_leak_global_to_target

target_state_leak_between_targets

shared_self_model_leak
并在 telemetry 中输出每项贡献。

新增阻塞性隔离场景（必须）
新增 YAML scenario：A/B 两个 target 交错输入（至少 30 轮），专门触发：

A 的 ledger promise → 不得影响 B 的 betrayal 判定

A 的 bond 变化 → 不得改变 B 的 bond

允许全局 mood/energy 若你定义为 global，则指标不能把它误判成“泄露”（需要定义清楚并体现在指标里）

若发现实现层面真的有泄露（允许修 bug）

定位：哪些 state 字段被错误地 global/shared

修复：明确归属（per-target / global），并补回归测试

验收

cross-target isolation scenario 在 baseline 必须可跑；并且能输出诊断报告

如果存在真实泄露，修复后 cross_target_interference 明显下降（设定最低目标：下降 20% 或解释为什么不能降）

D4: Live Integration Tests — Stop Skipping by Default
要做什么

把目前 “emotiond 未启动就 skip” 的 integration tests 变成更稳定的模式（二选一，推荐 A）：

A) 默认自动拉起 emotiond（推荐）

提供 test fixture：启动 emotiond（随机空闲端口）→ 等待 /health → 跑 tests → 关闭

加 --no-live 才禁用

B) 拆 CI job

unit tests 默认不跑 live

live tests 作为单独 job，保证稳定、可重复

验收

CI/本地默认命令不再出现不必要 skip（除非用户显式选择）

至少 8 个测试覆盖：启动/关闭、端口冲突、health timeout 的错误信息
