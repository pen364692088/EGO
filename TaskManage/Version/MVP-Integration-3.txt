emotiond 的 token 读取策略改为：

如何 rotate token

D2 稳定性（Hook 注入 & Trace）

Hook 写入 TOOLS.md 的 marker 块必须 严格限长（建议上限：2KB~4KB 文本；超过就截断并保留“trace 指针 + 时间戳 + 目标 id”）

Hook 注入内容只保留“本轮必需字段”：

target_id

最近 1 条 decision 摘要（短句 + 关键数值）

最新用户 affect 摘要（valence/arousal/confidence）

trace 文件位置（或 trace_id）

失败降级策略：

emotiond 不可用 / 超时 / 500 时：Hook 不得抛出导致对话失败的错误；应记录日志 + 跳过注入（或注入一段“emotiond unavailable”最短摘要）

Trace 轮转（避免磁盘滚雪球）：

traces 按 target_id 记录 JSONL 保持不变，但必须实现轮转策略之一：

按文件大小（例如 >5MB 切分）

或按日期（每天一个文件）

保留最近 N 份（例如 N=7 或 N=14），超出自动清理

清理动作必须安全：只动 integrations/openclaw/traces/ 下符合命名规则的文件

D3 User Affect v0（体验跃迁，但仍然低风险）

新增 schema（建议路径）：integrations/openclaw/schemas/user_affect.schema.json

输出结构（建议）：

{
  "valence": -1.0 ~ 1.0,
  "arousal": 0.0 ~ 1.0,
  "confidence": 0.0 ~ 1.0,
  "evidence": ["短证据1", "短证据2"]
}

判定逻辑原则：

置信度低（比如 <0.55）时：只输出 neutral/uncertain 风格，不改变高影响事件门控

affect 永远不触发 betrayal/rejection 等高影响事件（它只是一条“语气调制信号”）

增加测试：

典型语句（含 emoji、反讽、忙/冷处理等）覆盖：至少 12 条

测试重点：结构符合 schema + 范围合法 + 低置信度不越权
