

# 《双仓统一进度账 v1 设计稿》

## 0. 设计目标

建立一个**跨仓唯一总口径**，用来回答这 4 个问题：

1. 当前系统整体处于什么状态
2. 下一步到底允许做什么
3. 哪个阶段是“代码存在”，哪个是“已验证”
4. 哪个结论来自 EgoCore，哪个来自 OpenEmotion，哪个只是跨仓主线判断

本进度账**不废除**现有局部版本体系，而是统一解释它们：

* EgoCore 继续保留 `Phase / P0 / P1-A / P1-B / P1-C...`
* OpenEmotion 继续保留 `MVP11.5 / MVP12 / ... / MVP16`
* 主体最小可行主线继续保留 `WS-A / WS-B / WS-C/C1/C2...`

统一进度账负责把这三套编号映射成**同一种状态语义**。

---

## 1. 设计原则

### 1.1 只统一“状态语义”，不强行统一“编号”

禁止把三套局部编号硬并成一个编号体系。
原因：

* `P1-A/B/C` 主要描述宿主侧工作包与边界收口
* `WS-A/B/C` 主要描述主体最小可行能力主线
* `MVP11.5–16` 主要描述 OpenEmotion 的版本与验证路线

它们回答的是不同问题，不应直接画等号。 

### 1.2 总进度只能由统一进度账裁决

以后所有 README、handoff、状态文档、任务单，如果写“当前状态”“下一步”“是否允许推进”，必须先引用统一进度账，而不是直接拿某个局部编号当总状态。

### 1.3 单一权威源

任何状态只能有一个最终解释权来源。
允许 mirror / cache / shim，不允许双主。

### 1.4 阻塞优先于乐观叙事

一旦出现以下任一情况：

* `blocked`
* `disputed`
* `refuted`
* `code_exists_but_unverified`

则不得用“观测中”“已完成”“可进入下一阶段”覆盖。

### 1.5 统一进度账必须可用于 Gate

它不是摘要文档，而是要能直接驱动：

* Gate A：Contract
* Gate B：E2E
* Gate C：Integrity
  这些 Gate 已经在边界宪章中有正式定义。

## 2. 统一进度账的四条主轴

统一进度账 v1 采用四轴模型。

## 2.1 Host Axis（宿主轴）

回答：

* EgoCore 宿主、runtime、tools、governance 是否成熟
* 是否能稳定承载 OpenEmotion
* 是否处于稳链 / shadow / enforced 之前的哪个阶段

只看 EgoCore 本身的正式工程进度。

---

## 2.2 Subject Axis（主体能力轴）

回答：

* Minimum Viable Self 当前推进到哪里
* `WS-A / WS-B / WS-C/C1/C2...` 到了哪一步
* 当前最小主体能力缺口在哪里

这条轴直接服务于你现在的双核主体主线。
交接文档当前给出的基线是：

* `WS-A：完成`
* `WS-B：完成`
* `MVS v1：未完全完成`
* 原因是还差 `WS-C 至少完成 C1（三层记忆模型 v1)`。

---

## 2.3 Verification Axis（验证轴）

回答：

* 代码只是存在，还是已经被证明
* 哪个阶段通过了 main-chain / causal / replay / E2E 证明
* 哪个阶段只是路线规划，不是完成证明

这条轴优先承接 OpenEmotion 的 `MVP` 体系。
`VersionRoadmap.md` 目前清楚表达了 **MVP11.5 → MVP16** 是推荐推进主线。

---

## 2.4 Boundary Axis（边界完整性轴）

回答：

* 是否发生双主
* 是否把主体本体逻辑写回 EgoCore
* mirror / cache / shim 是否受控
* 是否违反 `WS-C` 起的硬边界

边界宪章已经把这条轴的硬规则写得很完整，包括：

* 权威源归属
* L0–L7 分层
* WS-C 起禁止把 memory / salience / consolidation / appraisal / reflection 写回 EgoCore
* 六问门禁
* Boundary Gate。

---

## 3. 状态语义枚举（统一状态字典）

统一进度账只允许使用以下状态值：

### 3.1 计划态

* `planned`：已明确计划，未开始
* `scoped`：范围已定义，未落代码
* `contract_defined`：契约/边界/schema 已定义

### 3.2 实现态

* `code_exists`：代码存在，但未完成验证
* `integrated`：已接入主链，但未完成验证
* `shadow_running`：已进入 shadow 观察
* `enforced`：已正式启用

### 3.3 证明态

* `verified_contract`：契约证明已完成
* `verified_e2e`：E2E 已证明
* `verified_integrity`：完整性已证明
* `verified_full`：A/B/C 全通过

### 3.4 风险态

* `blocked`：有明确阻塞，不允许继续晋级
* `disputed`：真相源冲突，禁止乐观口径
* `refuted`：已有证据推翻旧结论
* `deprecated`：已废弃
* `legacy_compat_only`：仅保留兼容，不再主线推进

## 4. 三套局部编号的正式职责

## 4.1 EgoCore 编号

命名空间：`EG_PHASE`

用途：描述宿主工程进度、治理壳进度、边界收口进度。
示例：

* `EG_PHASE:P1-A`
* `EG_PHASE:P1-B`
* `EG_PHASE:P1-C2`

禁止用途：

* 不得直接表达主体能力是否形成
* 不得直接表达 OpenEmotion 是否已验证

---

## 4.2 OpenEmotion 编号

命名空间：`OE_MVP`

用途：描述 OpenEmotion 内部版本主线与验证目标。
示例：

* `OE_MVP:11.5`
* `OE_MVP:12`
* `OE_MVP:16`

禁止用途：

* 不得直接表达双仓总体 readiness
* 不得替代 MVS 主体缺口判断

---

## 4.3 双核主体编号

命名空间：`SELF_WS`

用途：描述主体最小可行能力主线。
示例：

* `SELF_WS:A`
* `SELF_WS:B`
* `SELF_WS:C1`
* `SELF_WS:C2`

当前语义基线来自交接文档：

* `WS-A / WS-B` 属于已完成主体骨架段
* `WS-C/C1` 是当前主线缺口，用于让事件进入正式记忆。

## 6. 裁决规则

## 6.1 总状态如何产生

总状态 `overall_program_state` 不由单个版本号决定，而由四轴共同决定。

裁决优先级：

1. `boundary_axis`
2. `verification_axis`
3. `subject_axis`
4. `host_axis`

原因很简单：

* 边界错了，后面全是假推进
* 验证 blocked 了，不能靠“代码存在”继续推进
* 主体缺口没补齐，不能跳阶段
* 宿主再成熟，也不能替代主体本体完成

---

## 6.2 一票否决规则

出现以下任一情况，总状态自动不得高于 `blocked`：

* `boundary_axis = blocked | violated`
* `verification_axis = blocked | refuted`
* `subject_axis = disputed`
* 任一主线条目是 `code_exists` 却被外部文档写成 `completed`

---

## 6.3 “允许下一步”不看版本号，只看状态

例如：

* 即使 `OE_MVP = 16`，只要 `verification_axis = blocked`，也不能继续推进下一阶段
* 即使 `EG_PHASE = P3-D`，只要 `SELF_WS:C1 = disputed`，也不能进入 `C2`

---

## 7. 与 Gate 的关系

## 7.1 Gate A（Contract）

统一进度账必须记录：

* 功能归属
* 权威源
* 接口契约
* 是否双主
* 当前条目属于哪个命名空间

## 7.2 Gate B（E2E）

统一进度账必须记录：

* 是否真的接入主链
* 是否通过结构化接口联动
* 是否通过 replay / E2E / trace 证明

## 7.3 Gate C（Integrity）

统一进度账必须记录：

* shim / mirror / cache 是否登记
* 是否有到期版本
* 是否存在越界残留
* 是否把过渡态伪装成正式完成

这三类 Gate 要求与边界宪章完全一致。

## 8. v1 初始化填充建议

这是基于你当前文档集合的**最保守初始化**，不是最终结论。

### Host Axis

建议初始状态：

* `shadow_running` 或 `integrated`

原因：

* 交接文档把 EgoCore 侧 `P0 / P0.5 / P1-A / P1-B / P1-C1 / P1-C2 / 边界整改` 都写成已完成，并判断 EgoCore 已进入“宿主稳定期”。
* 但如果缺少独立验证报告，先不要直接写 `verified_full`。

### Subject Axis

建议初始状态：

* `disputed`

原因：

* 交接文档明确写 `WS-A：完成`、`WS-B：完成`，但 `MVS v1 未完全完成`，还差 `WS-C/C1`。
* 所以当前不能把 `WS-C/C1` 直接写成完成态。

### Verification Axis

建议初始状态：

* `blocked` 或 `planned_to_blocked_transition`

原因：

* `VersionRoadmap.md` 给的是 `MVP11.5 → MVP16` 的主线与目标定义，更适合当路线依据，不适合直接当完成证明。
* 如果后续有 handoff / roadmap state 明确写 blocked/refuted，应以那类状态文件覆盖路线文档。

### Boundary Axis

建议初始状态：

* `contract_defined`

原因：

* 边界宪章和交接文档已经把权威源、硬边界、L0–L7、WS-C 起的禁止项定义得很完整。 
* 但除非附上最新 regression scan，否则先别写 `verified_integrity`。

## 9. 必须同步的文件

统一进度账落地后，以下文件必须改成“引用统一进度账”，而不是各自口头解释：

* `OpenEmotion README`
* `OpenEmotion LATEST_HANDOFF.md`
* `OpenEmotion roadmap state`
* `EgoCore 当前状态说明`
* `OpenClaw 默认任务模板`
* `handoff 模板`

原则：
**谁写“当前状态/下一步”，谁就必须引用统一进度账。**

---

## 10. 更新规则

每次更新统一进度账，必须同时写明：

1. 更新人
2. 变更前状态
3. 变更后状态
4. 触发证据
5. 影响的下一步
6. 若有冲突，谁覆盖谁

禁止：

* 只改 README 不改统一进度账
* 只改 handoff 不改统一进度账
* 先口头说“已完成”再补账

---

## 11. 一句话裁决准则

> **局部编号描述局部推进；统一进度账裁决整体状态。**

## 12. 建议直接交付物

这版设计稿落地时，建议一次性交付 3 个文件：

1. `docs/PROGRAM_STATE_UNIFIED_SPEC.md`
   解释这套账是什么、怎么用

2. `docs/PROGRAM_STATE_UNIFIED.yaml`
   当前真实状态

3. `docs/PROGRAM_STATE_CHANGELOG.md`
   每次状态变更的证据链
