# Identity Stability and Expansion Governance

## 1. Purpose

This document defines how developmental expansion is governed so that
growth does not destroy continuity, auditability, or identity coherence.

## 2. Hard Constraints

The system must preserve:
- governance shell authority
- auditability
- replayability
- protected identity invariants
- non-bypass rules

## 3. Expansion Review

Before accepting major developmental expansion, the system must evaluate:
- continuity preservation
- invariant compatibility
- rollback feasibility
- governance safety
- branch fragmentation risk

## 4. Fragmentation Risks

Examples:
- incompatible self-model branches
- conflicting developmental goals
- unstable revision oscillation
- identity over-expansion without consolidation

## 5. Responses to Instability

When instability is detected:
- suspend promotion
- mark branch as provisional
- collect more evidence
- prefer reversible consolidation
- rollback if required

## 6. Required Metrics

Suggested metrics:
- continuity_break_count
- branch_fragmentation_rate
- unstable_expansion_count
- rollback_frequency
- invariant_violation_count
