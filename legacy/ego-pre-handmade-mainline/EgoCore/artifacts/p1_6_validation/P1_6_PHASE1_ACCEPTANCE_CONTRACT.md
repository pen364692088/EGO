## 1. 验收范围

Phase 1 验收覆盖以下核心能力：

1. **Telegram 入口** - 五类语义路由正常工作
2. **任务执行** - 任务创建、规划、执行、暂停、恢复
3. **Tool Runtime** - 工具真实调用，失败不伪装成功
4. **连续性** - memory + checkpoint + resume 链路完整
5. **恢复能力** - 程序重启后可恢复未完成任务
6. **安全边界** - 失败路径处理正确，高风险操作需确认

## 2. 通过标准

### 2.1 Telegram 五类入口

| 入口类型 | 触发示例 | 预期行为 |
|---------|---------|---------|
| chat | "你好" / "hello" | 友好问候回复 |
| question | "今天天气怎么样" | LLM 回答问题 |
| new_task | "帮我分析这个项目" | 创建任务并执行首步 |
| continue | "继续" / "go on" | 继续当前任务 |
| command | "/status" / "/tasks" | 执行对应命令 |

**验收标准**: 每类入口至少有 1 次成功调用证据

### 2.2 Telegram E2E 任务链路

至少 3 条真实 Telegram 任务链路：

1. **只读分析任务**
   - 触发: "读取 README 并总结"
   - 预期: 调用 file tool，返回总结
   - 证据: Telegram 消息截图 + 日志

2. **Tool 驱动任务**
   - 触发: "列出当前目录文件"
   - 预期: 调用 shell 或 file tool
   - 证据: 工具调用日志

3. **多轮连续性任务**
   - 触发: 创建任务 → 执行 → 暂停 → 继续
   - 预期: 任务状态正确流转
   - 证据: 状态变化日志

**验收标准**: 每条链路有完整的消息流 + 日志证据

### 2.3 Tool Runtime 真调用

| 工具 | 验证内容 | 证据要求 |
|-----|---------|---------|
| file | read/list 操作 | 日志显示 tool bus 调用 |
| shell | 执行命令 | 日志显示命令和输出 |
| python | 执行脚本（可选） | 日志显示执行结果 |

**失败案例**: 至少 1 个工具失败场景，验证失败不会被伪装为成功

**验收标准**: 至少 2 个工具被真实调用，有 1 个失败案例证据

### 2.4 Memory / Checkpoint / Resume 连续性

验证链路:
```
create task → execute step → save memory → save checkpoint 
→ pause → resume → continue execution
```

**证据要求**:
- memory 文件包含任务状态
- checkpoint 文件存在
- resume 后任务继续执行

**验收标准**: 完整链路证据，无断点

### 2.5 重启恢复

验证步骤:
1. 创建任务并执行到中间状态
2. 停止程序 (Ctrl+C)
3. 重新启动程序
4. 同一 chat 发送 "继续" 或 /resume
5. 任务恢复并继续执行

**证据要求**:
- 重启前任务状态落盘
- 重启后恢复成功
- 任务 ID 一致

**验收标准**: 至少 1 次真实重启恢复证据

### 2.6 失败路径与安全边界

| 场景 | 预期行为 |
|-----|---------|
| 无任务时 "继续" | 提示无任务，不误建任务 |
| tool 执行失败 | 任务状态 blocked/failed，非 completed |
| 高风险操作 | 需要确认，不自动执行 |

**验收标准**: 每个场景有测试证据
