

# EgoCore / OpenEmotion Agent Runtime

# OpenClaw 强制执行版 P1 任务单

## 主题：语义入口 + 自动首步执行 + continue/resume 绑定

## 任务名称

**P1 Semantic Runtime Upgrade：从命令式任务机升级为语义驱动单 Agent Runtime**

## 仓库

`https://github.com/pen364692088/EgoCore`

## 背景

P0 之后，系统虽然不再是最初那种“任何自然语言都误建任务”的状态，但当前仍明显停留在“命令式任务机”阶段，还没有达到“像 OpenClaw 那样自动判断、自动进入任务、自动推进第一步、自动继续上个任务”的体验。

当前暴露出的核心问题：

1. 普通聊天、任务请求、继续类消息没有正确分层
2. 消息之间疑似存在串线/污染，上一条输入会错误影响下一条
3. `/run` 执行时没有正确注入 runtime 上下文，反而向用户索要内部字段
4. 用户仍需手动 `/new` → `/run`，交互步骤太重
5. “继续 / 接着做 / 还有呢 / 上个任务怎么样了” 这类消息还不能自动绑定 active task

而你的第一阶段目标本来就不是复刻 OpenClaw 全家桶，也不是做复杂多 agent / heartbeat / DSL，而是先做一个 **Telegram 驱动、可恢复、记忆维持连续性、任务执行优先的独立 Agent Runtime**。

---

# 本次总目标

把 EgoCore 从：

**命令式 task shell**

升级为：

**语义驱动的单 Agent Runtime**

达到以下体验：

* 用户发“你好”“在吗” → 正常聊天，不建任务
* 用户发“帮我检查 EgoCore 仓库现在还有什么核心问题” → 自动识别为任务请求，自动建任务，自动规划，自动执行第一步
* 用户发“继续”“接着做”“上个任务怎么样了”“还有呢” → 自动绑定最近 active task 或可恢复任务
* 用户不必在大多数情况下手动 `/new` 再 `/run`
* `/run` 仍保留，但退化为显式控制补充，而不是主要入口

---

# 强制边界

## 1. 本次不允许做的事

本次只做语义入口闭环，不允许借机扩成平台工程：

* 禁止引入多 agent 编排
* 禁止引入 heartbeat / cron / auto-resume 全套
* 禁止引入复杂 workflow DSL
* 禁止做大型 web UI / dashboard
* 禁止重写整套 runtime
* 禁止新增大量花式命令来掩盖语义入口缺陷

这些都不属于第一阶段主目标。当前主轴仍应是 Task Runtime，由 Telegram 作为控制入口、记忆提供连续性、工具提供执行能力、checkpoint 提供恢复能力。

## 2. 本次禁止假语义

以下情况一律视为未完成：

* 只是把规则写进 prompt，但代码路由没改
* 只是新增 `/continue` 命令，而自然语言“继续”仍不生效
* 只是自动创建任务，但不会自动执行第一步
* 继续类消息只会回帮助文案，不会绑定已有任务
* runtime 仍向用户索要 `taskgoal/currentstep/...` 等内部字段
* 仍存在上一条消息影响下一条消息的串线污染

## 3. 本次必须以真实使用体验为标准

只要 Telegram 真实对话体验仍需要用户机械地：

`/new` → `/run` → `/report`

才能完成简单任务，就不能算 P1 完成。

# 必做任务拆解

## T1. 实现语义入口分类器

### 目标

每条 Telegram 消息先做语义分类，再决定进入哪条链路。

### 必须支持的最小分类

至少分成 5 类：

1. **chat**

   * 问候、寒暄、状态确认
   * 如：你好、在吗、早上好、你还在吗

2. **question**

   * 普通问答 / 咨询 / 解释
   * 如：为什么 /run 之前不生效？
   * 如：这个项目方向对吗？

3. **new_task**

   * 明确任务请求
   * 如：帮我检查 EgoCore 仓库现在还有什么核心问题
   * 如：读取 README 并总结关键点
   * 如：帮我生成任务单

4. **continue_task**

   * 继续、恢复、追问当前任务
   * 如：继续
   * 如：接着做
   * 如：上个任务怎么样了
   * 如：还有呢
   * 如：继续上一个任务

5. **command**

   * `/new`
   * `/run`
   * `/report`
   * `/status`
   * `/resume`
   * 其他显式命令

### 强制要求

* 分类必须先于 task create
* 普通自然语言不得直接默认 create task
* continue 类消息不得落到普通聊天或新任务创建
* 命令仍然保留最高优先级

### 验收标准

* “你好” → chat
* “为什么刚刚会误建任务” → question
* “帮我检查这个仓库还有什么大问题” → new_task
* “继续” → continue_task
* `/run` → command

---

## T2. 修复消息串线 / 上下文污染

### 目标

确保当前输入只基于当前消息和受控会话状态处理，不会错误复用上一条自然语言内容。

### 必做项

* 检查 Telegram adapter 到 router 的 message passing
* 检查 pending input / last user message / current task draft 是否被错误复用
* 检查是否存在：

  * 上一条自然语言被残留在会话缓存
  * 新消息未正确覆盖输入
  * command handler / nl handler 共享脏状态
* 对 router 引入最小可观察诊断字段：

  * current_message
  * classified_intent
  * matched_task_id
  * source_of_context

### 强制要求

必须能解释清楚为什么之前：

* 先发“你好”
* 再发“还有其他问题吗”
* 却创建了 objective = “你好”

如果根因没有被明确定位和修复，不能宣称完成。

### 验收标准

连续发送 5 条不同消息，不得出现上一条内容污染下一条分类或任务目标。

---

## T3. 实现自动新任务入口

### 目标

对于明确的新任务请求，系统自动：

**识别 → 创建任务 → 规划 → 自动执行第一步 → 回复阶段结果**

### 必做项

当消息被分类为 `new_task` 时：

1. 创建 task
2. 保存 objective
3. 注入相关 memory / context
4. 生成最小计划
5. 自动执行第一个安全步骤
6. 返回：

   * task id
   * 当前状态
   * 已完成内容
   * 当前正在做什么
   * 下一步建议

### 强制要求

* 不再强依赖用户先输入 `/new`
* 自动首步执行必须是真执行，不是只建任务不动
* 自动首步执行只允许执行安全且低破坏性的第一步

  * 例如读取文件、列目录、分析文本、给出计划
* 高风险操作仍必须确认后再做

### 验收标准

用户发送：
“帮我检查 EgoCore 仓库现在还有什么核心问题”

预期：

* 自动创建 task
* 自动执行第一步
* 直接返回阶段性结果
* 用户无需额外发 `/new` `/run`

---

## T4. 实现 continue / resume 自然语言绑定

### 目标

让“继续类消息”优先绑定现有任务，而不是要求命令式操作。

### 绑定策略

当分类为 `continue_task` 时，按以下优先级绑定：

1. 当前 active running / blocked / paused task
2. 最近一次未完成任务
3. 最近一次可恢复任务（checkpoint / task memory 可恢复）
4. 若都没有，再提示用户没有可继续任务

### 必做项

* 引入 active task resolver
* `/resume` 与自然语言“继续”共享一套恢复逻辑
* “还有呢”“接着做”“继续上一个任务”都应落入同一语义分支
* 若绑定成功：

  * 自动恢复上下文
  * 自动执行下一步或给出可执行建议
* 若绑定失败：

  * 明确告诉用户没有可继续任务
  * 不要创建一个新的“继续”任务

### 强制要求

* continue 类消息绝不能误建新任务
* continue 类消息绝不能回到通用 help 壳子
* continue 绑定后必须基于 task memory / checkpoint，而不是重新从零猜

这也符合你第一阶段里对 `/resume`、task memory、checkpoint 恢复的原始要求。 

### 验收标准

先完成一个未完任务，再发送：

* “继续”
* “接着做”
* “上个任务怎么样了”

三者都应绑定同一个最近未完成任务。

---

## T5. 修复执行上下文注入

### 目标

executor 必须从 runtime 获取上下文，不能再向用户暴露内部协议字段。

### 必做项

每次执行 step 前，runtime 必须显式注入：

* task objective
* current step
* completed steps summary
* step index / total steps
* available tools
* relevant memory summary
* latest checkpoint summary
* current execution mode

### 强制要求

禁止再次出现这种输出：

* “请提供 taskgoal”
* “请提供 currentstep”
* “请提供 stepnumber”
* “请提供 totalsteps”

这些字段是 runtime 内部状态，不是用户输入协议。

### 验收标准

`/run` 或自动首步执行时，executor 输出必须基于已有 task state 正常工作，不得再向用户索要内部字段。

---

## T6. 建立“自动做”和“先确认”边界

### 目标

做到像 OpenClaw 那样尽量自动，但不越过安全边界。

### 必做项

把新任务执行分成两类：

### A. 可自动首步执行

* 读取文件
* 列目录
* 总结内容
* 代码静态分析
* 生成计划
* 输出建议
* 简单只读检查

### B. 必须确认后执行

* 写文件
* 删除文件
* git push / force 操作
* 系统服务重启
* 联网调用高风险外部接口
* 可能造成不可逆修改的命令

### 强制要求

* P1 的目标是减少交互步骤，不是取消安全边界
* 自动首步执行仅限安全只读或低风险步骤
* 一旦需要确认，必须明确告诉用户原因

### 验收标准

* “读取 README 并总结” → 自动执行
* “帮我删除这个目录” → 先确认
* “帮我修改并提交代码” → 可先分析和规划，但写入/提交前必须确认

---

## T7. 优化回复形态，去掉机械感

### 目标

用户感受接近 agent，而不是表单机。

### 必做项

按意图提供不同回复模式：

### chat

自然短回复，不提 task unless relevant

### question

直接回答，必要时可补“若你要我继续实做，可直接说一句”

### new_task

回复结构：

* 已识别为任务
* 已自动开始
* 当前结果
* 下一步

### continue_task

回复结构：

* 已绑定哪个任务
* 当前进展
* 刚刚继续执行了什么
* 下一步

### command

命令式简洁响应

### 强制要求

* 不要把所有自然语言都回复成“请使用 /new”
* 不要把所有执行结果都回复成僵硬模板
* 不要在普通聊天里突然抛 task id
* 回复风格应体现“我理解了你是在聊天 / 问问题 / 给任务 / 继续任务”

---

## T8. 与现有命令共存，而不是互相打架

### 目标

语义入口是主入口，命令入口是显式控制补充。

### 必做项

保留现有命令：

* `/new`
* `/run`
* `/status`
* `/tasks`
* `/resume`
* `/pause`
* `/retry`
* `/abort`
* `/report`
* `/memory`

但重新定义地位：

* 普通用户路径：自然语言驱动
* 专业控制路径：命令驱动
* 两者底层共用同一套 task runtime 和 resolver

### 强制要求

不能做成两套完全分裂的逻辑：

* 自然语言是一套
* 命令是另一套

必须共享：

* task store
* active task resolver
* execution engine
* memory/checkpoint 恢复链

1. 普通聊天不建任务

**“P1 部分完成，未达语义闭环标准”**
