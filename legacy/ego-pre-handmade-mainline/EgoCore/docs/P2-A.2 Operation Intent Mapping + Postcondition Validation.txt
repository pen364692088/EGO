## 任务背景

根据真实 Telegram 测试，当前系统存在严重执行偏差：

### 已确认问题 1：目录目标被错误忽略

用户请求：

* `帮我看看/home/moonlight/Project/Github/MyProject/EgoCore/docs里有哪些文件`
* `帮我看看/home/moonlight/Project/Github/MyProject/EgoCore/docs/里有哪些文件`

实际结果却总是列出：

* `/home/moonlight/Project/Github/MyProject/EgoCore`

也就是项目根目录，而不是目标 `docs/` 目录。

### 已确认问题 2：创建文件请求根本没执行创建

用户请求：

* `帮我在/home/moonlight/Project/Github/Test/里创建一个test.md文件`

实际结果仍然是列出 EgoCore 根目录内容，**没有创建文件**。

### 已确认问题 3：错误结果被标成 completed

虽然执行结果明显不符合用户目标，但系统仍返回：

* `状态: completed`
* `任务已完成`

这说明当前 runtime 只判断“某个工具调用有没有成功返回”，没有判断“返回结果是否满足原始任务目标”。

---

## 根因判断

### 根因 A：默认执行器没有正确抽取“操作类型 + 目标路径”

当前默认执行器对“文件/目录”类请求的语义映射过于粗糙，导致：

* 目录查看请求没正确提取目标目录
* 创建文件请求没正确映射为 `write` / `mkdir`
* 最终错误退化成固定的目录 list 行为

### 根因 B：目录 listing 分支存在错误默认路径

当前目录处理逻辑存在“落不到明确路径时默认列 `.`”的问题，导致很多请求都退化成列当前工作目录。

### 根因 C：缺少 postcondition validation

当前只校验：

* 工具有没有成功执行

没有校验：

* 是否操作了正确目标路径
* 是否完成了正确操作类型
* 是否满足了用户原始目标

所以会出现：

* 实际做错事
* 但返回 completed

---

# 本次总目标

把系统从：

**“能识别成任务，但常常执行错动作、还假装成功”**

修成：

**“能正确识别操作意图、命中正确路径、真实验证结果、失败不伪装完成”**

本次完成后，至少必须满足：

* 看目录 → 真看用户指定目录
* 读文件 → 真读用户指定文件
* 创建文件 → 真创建用户指定文件
* 创建目录 → 真创建用户指定目录
* 如果执行结果与目标不一致 → 不能 completed
* `/report` 和最终回复能明确说明执行目标、实际动作、验证结果

# 必做任务拆解

## T1. 建立操作意图映射层

### 目标

在默认执行器真正调用 tool 之前，先把用户请求解析成明确操作。

### 必做项

至少支持以下最小操作集合：

* `list_dir(path)`
* `read_file(path)`
* `write_file(path, content?)`
* `mkdir(path)`
* `exists(path)`

### 识别范围

至少覆盖以下中文表达：

* 看看 xxx 里有哪些文件
* 列出 xxx 目录内容
* 读取 xxx 文件
* 打开 xxx 文件
* 在 xxx 里创建一个 xxx 文件
* 在 xxx 下创建目录 xxx
* 检查 xxx 是否存在

### 强制要求

* 操作类型必须显式解析，不允许继续靠“包含 文件/目录 字样”粗暴分流
* 解析结果必须结构化，例如：

  * `operation_type`
  * `target_path`
  * `target_name`
  * `content_if_any`

### 验收标准

对于下面请求，解析结果必须正确：

* `帮我看看/home/.../docs里有哪些文件` → `list_dir(/home/.../docs)`
* `帮我在/home/.../Test/里创建一个test.md文件` → `write_file(/home/.../Test/test.md)`

---

## T2. 修复路径抽取逻辑

### 目标

用户说哪个路径，就操作哪个路径，不允许退化到 `.`。

### 必做项

支持正确抽取以下路径形式：

* `/home/moonlight/.../docs`
* `/home/moonlight/.../docs/`
* 带中文前后缀的绝对路径
* 带“里/下/里面/目录里”的句式
* 文件路径与目录路径区分

### 强制要求

* 目录 listing 逻辑禁止默认硬编码 `"."`
* 只有在用户明确要求“当前目录”时，才允许使用当前目录
* 若路径无法可靠解析，应返回 `blocked / validation_error`，而不是胡乱执行

### 验收标准

对你这次的 `docs` 场景，实际调用必须命中：

* `/home/moonlight/Project/Github/MyProject/EgoCore/docs`

而不是根目录。

---

## T3. 为创建文件/写文件补真实执行链

### 目标

“创建文件”类请求不能再误落到目录 listing。

### 必做项

当识别为 `write_file` 时：

1. 解析目标目录
2. 拼接目标文件路径
3. 调用 `file_tool.write`
4. 必要时自动创建父目录或明确报错
5. 执行后立即做 `exists` 校验
6. 只有校验通过才可标记成功

### 强制要求

* “创建 test.md 文件”必须走写文件链
* 不允许用列目录结果代替创建成功
* 若当前系统不支持无内容创建，必须明确策略：

  * 创建空文件
  * 或写默认内容
  * 或要求用户提供内容
    但不能假装成功

### 验收标准

请求：

* `帮我在/home/moonlight/Project/Github/Test/里创建一个test.md文件`

结果必须满足：

* 文件真实存在
* 回复中明确显示创建路径
* 不再返回 EgoCore 根目录列表

---

## T4. 为目录查看补真实目标校验

### 目标

list_dir 不仅要执行，还要验证执行的是不是用户指定目录。

### 必做项

目录查看完成后，必须校验：

* 实际操作路径 == 解析出的目标路径
* 返回结果来自该目录
* 若目录不存在，返回正确错误
* 若路径被安全策略拦截，返回明确原因

### 强制要求

* 不能只要 list 成功就 completed
* 如果目标是 `/docs`，结果却来自根目录，必须视为失败
* 校验失败时必须进入 `failed` 或 `blocked`

### 验收标准

`看 docs 里有哪些文件` 场景，若实际路径不是 `docs`，任务不得 completed。

---

## T5. 建立 postcondition validation

### 目标

把“工具调用成功”和“任务目标达成”分开。

### 必做项

为至少以下操作建立后置校验：

* `list_dir`

  * 验证实际路径
  * 验证返回项是否来自目标目录

* `read_file`

  * 验证实际读取路径
  * 验证读取内容存在

* `write_file`

  * 验证目标文件存在
  * 必要时验证内容可读回

* `mkdir`

  * 验证目录已存在

### 强制要求

* postcondition 失败必须改变结果状态
* 不允许“工具成功了所以任务成功”
* postcondition 必须写入：

  * unified result
  * task memory
  * `/report`
  * event log

### 验收标准

任何执行结果若与用户目标不一致，都不能显示 completed。

---

## T6. 统一“意图达成失败”的失败分类

### 目标

让“执行了，但没做到用户要的事”有明确归因。

### 建议新增 failure class

例如：

* `INTENT_MISMATCH`
* 或 `POSTCONDITION_FAILED`

### 必做项

当出现以下情况时进入该分类：

* 目标路径解析错
* 实际操作路径错
* 操作类型错
* 写文件请求却执行成 list
* 列目录请求却列错目录

### 强制要求

* 该失败分类必须进入：

  * task status
  * `/report`
  * `/resume`
  * task memory
  * checkpoint

### 验收标准

这次测试暴露的问题，必须被标记为明确 failure class，而不是 completed。

---

## T7. 修正用户可见回复

### 目标

让用户一眼看出系统到底做了什么、验证了什么。

### 必做项

成功回复中至少包含：

* 识别到的操作类型
* 目标路径
* 实际执行路径
* postcondition 是否通过

失败回复中至少包含：

* 失败分类
* 为什么没完成
* 当前 blocker
* 下一步建议

### 强制要求

不能再出现这种情况：

* 用户要创建文件
* 回复却是一坨目录列表
* 最后还说任务已完成

### 验收标准

用户看到回复后，能明确判断：

* 是不是做对了目标动作
* 如果没做对，是卡在哪一步
