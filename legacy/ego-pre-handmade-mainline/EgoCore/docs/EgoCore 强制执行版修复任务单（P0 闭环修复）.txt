# EgoCore 强制执行版修复任务单（P0 闭环修复）

## 任务名称

**EgoCore Telegram Runtime P0 修复：禁止误建任务、修复 /run、打通真实最小闭环**

## 仓库

`https://github.com/pen364692088/EgoCore`

## 任务背景

当前 Telegram 运行链路暴露出严重偏差，已经影响基础可用性：

1. 普通自然语言消息会被错误当成新任务创建
   例：发送“在吗”会直接创建 task

2. `/run` 在 Telegram 侧未正确接入，表现为空转或无效

3. 命令入口、自然语言入口、任务执行入口没有正确分流

4. 任务执行链存在“假成功/占位成功”风险
   即步骤未真实完成，但状态被推进为完成

5. memory / checkpoint / event log 模块虽然存在，但未真正接入主执行链

## 总目标

把 EgoCore 从“原型脚手架”修到“最小可真实使用闭环”。

本次只做 **P0 修复**，不扩功能，不做大重构，不引入新框架。
目标是让 Telegram 入口至少具备以下能力：

* 普通聊天不会误创建任务
* `/new` 能明确创建任务
* `/run` 能真实执行当前任务下一步
* `/report` 能正确反映任务进度
* 未执行成功的步骤不能伪装成成功
* 执行过程有最小事件记录、checkpoint、恢复基础

# 必做任务拆解

## T1. 修复 Telegram 命令注册缺失

### 目标

确保 `/run` 真正被 Telegram bot 注册并可执行，不再出现命令发出去却未进入处理逻辑的情况。

### 必做项

* 检查 Telegram bot 命令注册列表
* 把 `run` 正式加入注册
* 检查 `/report`、`/resume`、`/pause`、`/retry` 等命令是否也都正确注册
* 增加 unknown command fallback

  * 用户输入未知命令时，必须返回明确提示，而不是静默无响应

### 验收标准

* Telegram 中发送 `/run`，能进入实际处理逻辑
* Telegram 中发送 `/not_exist_cmd`，会收到明确错误提示和帮助信息

---

## T2. 修复自然语言入口误建任务

### 目标

普通聊天不能再默认进入“创建任务”逻辑。

### 强制策略

**P0 阶段采用最稳妥策略：只有显式任务入口才允许创建任务。**

### 必做项

将入口分成至少三类：

1. **命令消息**

   * `/new`
   * `/run`
   * `/report`
   * `/tasks`
   * `/help`
   * 其他命令

2. **普通聊天消息**

   * 问候，如“在吗”“hello”
   * 状态确认，如“你还在吗”
   * 普通对话，如“这个项目怎么回事”

3. **任务型自然语言**

   * P0 阶段不要直接自动创建任务
   * 对于疑似任务请求，只提示用户使用 `/new`
   * 例如回复：

     * “这看起来像任务请求，请使用 `/new 任务内容` 创建任务”

### 明确要求

* `handle_natural_language()` 不能再直接调用“新建任务”
* 禁止任何普通自然语言默认落入 `_handle_new()`

### 验收标准

* 发“在吗” → 返回正常聊天/帮助提示，不创建 task
* 发“帮我整理一个 roadmap” → 不直接创建 task，而是提示用 `/new`
* 发 `/new 帮我整理一个 roadmap` → 正常创建 task

---

## T3. 修复 `/run` 的真实执行语义

### 目标

`/run` 必须执行“当前活动任务的下一步”，而不是空转、假推进或不明确行为。

### 必做项

* 明确定义 `/run` 的行为：

  * 若有当前 active task，则执行下一步
  * 若没有 active task，则返回明确提示
  * 若任务已完成，则提示任务已完成
  * 若当前步骤失败，则返回失败原因，不得自动伪装为完成

* 检查 task state 更新逻辑：

  * step 开始
  * step 完成
  * step 失败
  * task 完成
  * task 阻塞

### 验收标准

* `/new xxx` 后 `/run` 能推进 1 步
* `/report` 中 progress 会真实变化
* 没有 active task 时 `/run` 会提示，而不是沉默
* 执行失败时，状态变为 failed / blocked，而不是 completed

---

## T4. 去掉“占位成功 / 假成功”

### 目标

任务执行必须“真实成功才标记成功”，不允许因为 LLM/tool 不可用就默认写成“步骤完成”。

### 必做项

检查 runtime executor 逻辑，重点修复：

* 若只是生成占位文本，不能写成 completed
* 若 LLM 不可用：

  * 状态应为 blocked 或 failed
  * 必须记录明确原因
* 若工具未命中：

  * 可以返回 unsupported / no_executor
  * 但不能写成成功完成
* 若真实完成：

  * 才能推进 step index / task progress

### 强制要求

把“看起来像成功”的 fallback 文本与“真实成功”彻底分离。

### 验收标准

构造一个当前系统无法真实执行的任务：

* `/new 帮我分析这个远程网站并自动部署`
* `/run`
* 如果当前能力不支持，必须显示 blocked/unsupported
* 绝不能显示“步骤完成”

---

## T5. 把 event log / checkpoint / memory 接入主链

### 目标

不是模块存在，而是主链真的调用。

### 必做项

至少在以下节点写入事件：

* task created
* step started
* step completed
* step failed
* task completed
* task aborted / paused / resumed

至少在以下节点保存 checkpoint：

* task 创建后
* 每次 step 完成后
* task 状态变化时

至少实现最小 memory 写入：

* task objective
* latest status
* step summary
* final summary（完成时）

### 说明

P0 不要求做复杂认知记忆系统。
只要求把“任务相关最小记忆”真正落盘并可恢复。

### 验收标准

* 完成一个任务后，能从 memory/checkpoint 看到实际记录
* bot 重启后，活动任务状态不会完全丢失
* `/report` 不依赖内存中临时对象独存

---

## T6. 做真实 E2E 验证，不允许只跑单测

### 目标

必须用真实 Telegram 对话证明主链修通。

### 必做项

至少验证以下 6 条：

### 场景 A：普通聊天不误建任务

发送：

* `在吗`

预期：

* 正常回复
* 不创建 task

### 场景 B：提示用户使用显式任务入口

发送：

* `帮我整理一个 roadmap`

预期：

* 不自动建 task
* 提示使用 `/new`

### 场景 C：显式创建任务

发送：

* `/new 帮我整理一个 roadmap`

预期：

* 创建 task 成功
* 返回 task id / 状态 / 步骤信息

### 场景 D：执行下一步

发送：

* `/run`

预期：

* 实际执行下一步
* `/report` 中进度变化

### 场景 E：未知命令

发送：

* `/abcxyz`

预期：

* 明确提示未知命令
* 返回帮助信息

### 场景 F：失败不伪装成功

创建一个当前不支持的任务并执行：

预期：

* 返回 blocked/failed/unsupported
* 不推进为 completed
