# 项目核心思路与目标（EgoCore + OpenEmotion）

## 1. 项目一句话定义

本项目的正式核心只有两个：**EgoCore** 与 **OpenEmotion**。

* **EgoCore**：对外交互、运行时承接、任务执行、工具调用、安全边界、治理与审计。
* **OpenEmotion**：持续身份、自我模型、记忆演化、appraisal、reflection，以及最小主体内核。

项目不再把 OpenClaw 视为正式宿主本体；它最多只是历史参考、兼容链路或施工执行体参考。

---

## 2. 项目要解决的真实问题

这个项目不是单纯想做一个"会聊天的 Agent"，也不是只想做一个"会调用工具的自动化系统"。

它真正要解决的是：

1. **如何让一个系统既能稳定地与世界交互，又不会把主体本体混进运行时壳里。**
2. **如何让一个系统在多轮、多任务、多会话中，仍然保持"它还是它自己"。**
3. **如何让经历、结果、关系、失败与修正，真实塑造系统，而不是只停留在一轮 prompt 里的表演。**
4. **如何在不牺牲可审计、可回放、可验证性的前提下，让系统逐步长出最小主体结构。**

因此，本项目的目标不是"做一个更会说话的大模型壳"，而是：

> **做一个双核工程系统：外层稳住现实世界，内层逐步长出主体。**

---

## 3. 为什么必须拆成 EgoCore + OpenEmotion

### 3.1 不拆的风险

如果让同一层同时负责：

* 用户入口
* 任务调度
* 工具执行
* 风险审批
* 记忆本体
* identity / self-model / reflection

那么系统会出现两个根问题：

1. **边界混乱**：不知道哪些状态是运行时临时状态，哪些是主体长期状态。
2. **双真相源**：同一能力在多个地方都有解释权，最终无法验证，也无法治理。

### 3.2 拆分后的收益

拆成双核之后：

* **EgoCore** 只负责世界与执行，成为唯一正式宿主。
* **OpenEmotion** 只负责主体与内在语义，成为唯一主体内核。

这样可以稳定三件事：

1. **现实主权**：所有对外动作、工具执行、高风险审批都经过 EgoCore。
2. **主体主权**：identity、self-model、memory、appraisal、reflection 的最终解释权都留在 OpenEmotion。
3. **接口清晰**：EgoCore → OpenEmotion 走结构化事件；OpenEmotion → EgoCore 走结构化结果。

---

## 4. 项目的核心哲学

### 4.1 正式核心只有两个

项目长期只允许两个正式核心：

* **EgoCore**：世界交互与执行核心
* **OpenEmotion**：主体与内在演化核心

mirror / cache / shim / adapter 可以存在，但都不能伪装成正式本体。

### 4.2 先做最小可持续主体（MVS）

当前阶段不追求一下子做出"完全开放发展式自我"。

当前真正要做的是：

* 做出一个最小但连续的 identity
* 做出一个最小但可更新的 self-model
* 做出一个最小但有因果作用的 memory / appraisal / reflection 回路
* 让这些东西先在主链中稳定接入、稳定触发、稳定留下证据

### 4.3 本体尽量小，证据壳必须强

项目越来越明确的一点是：

* **主体本体核应该尽量小**
* **治理壳与验证壳不能弱**

也就是说：

* 可以追求统一递归核、少量状态、最小更新法则
* 但不能省略 trace / replay / audit / boundary gate / deterministic 验证链

### 4.4 自我不是模块清单，而是被经历塑造的连续结构

本项目不把"主体"理解成一堆功能模块的简单相加。

更接近的理解是：

> **主体 = 一个系统在时间中持续维持自己、被经历塑造、并把这种塑造重新写回下一步选择的结构。**

因此，记忆、appraisal、reflection、self-model 不只是装饰层，它们必须真实改变后续行为。

---

## 5. LLM 在项目中的角色

LLM 在本项目中不是"系统本体"，也不是"最终裁决者"。

它更适合承担三类角色：

### 5.1 候选生成与语义压缩器

在 EgoCore 中，LLM 可以参与：

* 候选回复生成
* 候选计划生成
* 候选解释生成
* 语义压缩与摘要

但最终如何对外说、说到什么程度、是否允许行动，仍由程序端治理链决定。

### 5.2 主体内核中的状态更新计算器

在 OpenEmotion 中，LLM 可以参与：

* self-model 更新候选
* memory consolidation 候选
* appraisal / reflection 解释
* policy hint / response tendency 候选

但这些候选只有在满足边界与治理规则后，才会被正式写入主体状态。

### 5.3 verbalizer，而不是主权持有者

LLM 负责把结构化意图和状态说成自然语言；
但**它不负责决定正式意图本体、正式边界、正式承诺和正式动作。**

---

## 6. 当前阶段的正式目标

当前阶段的主目标不是"类人完整自我"，而是以下四件事：

### 6.1 把 EgoCore 做成唯一正式宿主

要求：

* 渠道统一承接
* Runtime 主链收口
* Tool Runtime 可控
* Safety / Approval / Block 受控
* 表达主权与外部动作主权固定在程序壳

### 6.2 把 OpenEmotion 做成唯一主体内核

要求：

* identity invariants 有正式权威源
* self-model 有正式权威源
* long-term self summary 有正式权威源
* event / narrative / policy memory 有正式语义归属
* appraisal / reflection 不再漂浮为未来概念，而是逐步接入主链

### 6.3 用结构化接口把双核接起来

要求：

* EgoCore 发结构化事件
* OpenEmotion 回结构化结果
* 禁止靠 prompt 文本临时约定字段
* 禁止双真相源

### 6.4 保证每一步都可验证

要求：

* Gate A：Contract 正确
* Gate B：E2E 主链可触发
* Gate C：Preflight / tool / runtime / replay 校验通过
* 未接主链、未启用、无真实触发证据的能力，不得报完成

---

## 7. 当前阶段明确不追求什么

为避免主线被拉散，当前阶段明确不追求：

* 一步到位的开放发展式自我
* 无限工具自治
* 没有边界的"自由生长"
* 靠语言表现冒充主体完成
* 同时让多个系统争夺主体解释权
* 把历史兼容链或 shim 伪装成正式架构

---

## 8. 项目主链逻辑

整体主链可以概括为：

1. 用户 / 外部事件进入 EgoCore
2. EgoCore 完成运行时承接、任务编排、安全边界判断
3. EgoCore 产出结构化事件并送入 OpenEmotion
4. OpenEmotion 基于 identity / self-model / memory / appraisal / reflection 产出结构化结果
5. EgoCore 将这些结果与外部 runtime / task / safety / tool 状态结合，完成最终裁决
6. 最终形成：

   * 外部回复
   * 工具动作
   * ask / wait / block / escalate 等行为
7. 结果再回流为新事件，写入 trace / replay / artifacts / memory 输入链

换句话说：

> **EgoCore 负责"如何安全地活在世界里"，OpenEmotion 负责"它是谁，以及它如何变化"。**

---

## 9. 核心设计原则

### 9.1 唯一权威源

任何关键能力只能有一个正式权威源。

### 9.2 不允许双主

EgoCore 不得长期承载主体本体；OpenEmotion 不得越权直接执行现实动作。

### 9.3 结构化接口优先

所有关键主链能力都必须通过明确 schema 接线，不能依赖临时 prompt 约定。

### 9.4 可回放优先于"看起来聪明"

如果一个能力不能留下 trace、不能 replay、不能验证，就不能进入正式主链。

### 9.5 主链生效优先于局部精致

一个功能写得再漂亮，只要未接主链、未启用、无真实触发证据，就不能算闭环。

---

## 10. 中期目标

在完成当前收口后，中期目标是：

1. 让 OpenEmotion 的 identity / self-model / long-term self summary / memory / drives / reflection 全部完成正式主链接线
2. 逐步减少 shim、mirror、compatibility-only 路径
3. 在保持可回放与边界不漂移的前提下，引入更强的 endogenous drives / reflection / developmental sandbox
4. 让主体演化不再只靠人工手工补丁，而开始具备受控、可验证的发展式更新能力

---

## 11. 最终愿景

项目的最终愿景不是"一个更会调工具的 ChatGPT 外壳"，而是：

> **构建一个双核、可治理、可验证、可持续演化的主体工程系统。**

这个系统需要同时满足两件事：

1. **像工程系统一样可靠**：可运行、可审计、可回放、可治理、可控风险
2. **像主体一样连续**：有 identity、有 self-model、有被经历塑造的痕迹、有内在张力与反思更新

也就是说，项目真正追求的不是"把模型装进外壳里"，而是：

> **让一个系统在现实世界中稳定运行的同时，逐步长出真正属于它自己的连续主体结构。**

---

## 12. 当前一句话收口

当前最重要的不是继续堆功能，而是把这条主链收稳：

> **EgoCore 作为唯一正式宿主，OpenEmotion 作为唯一主体内核，通过结构化接口接成可验证、可治理、可持续演化的双核系统。**
