## 一、总目标

为 **OpenEmotion** 构建一个 **独立、轻量、可闭环运行** 的 Agent Runtime。
该 Runtime **不依附于 OpenClaw 作为宿主运行**，但允许在架构、模块划分、任务执行、工具调用、记忆、checkpoint、Telegram 接入、配置组织等方面 **参考 OpenClaw 的成熟设计思路**。

第一阶段目标不是做通用平台，也不是复刻 OpenClaw，而是先交付一个 **真实可运行、可恢复、可扩展的最小闭环程序**。

---

## 二、第一阶段必须达成的闭环

第一阶段必须至少形成以下真实闭环：

用户在 **Telegram** 发送任务
→ 系统识别任务并创建 task
→ 读取相关记忆与配置
→ 基于 prompt + model 配置进行规划
→ 调用工具执行
→ 保存 task state / memory / event log / checkpoint
→ 返回 Telegram 回复
→ 用户可继续发 `/status` `/resume` `/report`
→ 程序重启后仍能恢复最近任务基本状态
→ 后续可继续平滑接入 OpenEmotion 高层能力

---

## 三、第一阶段核心能力

### 必须包含

1. **Telegram 交互**
2. **任务执行内核**
3. **良好记忆**
4. **基础工具系统**
5. **checkpoint / 恢复**
6. **完整配置层**
7. **OpenEmotion bridge 预留接口**

### 明确禁止跑偏

第一阶段**不要做成大而全平台**，禁止以下方向：

* 不做复杂多 agent 编排
* 不做复杂 cron / heartbeat / auto-resume 全家桶
* 不做大型 dashboard / web UI
* 不做完整插件生态
* 不做复杂 router / workflow DSL
* 不做平台感驱动的过度抽象
* 不做 OpenClaw 全量复刻
* 不为了“以后也许有用”提前堆太多基础设施

---

## 四、核心原则

### 1. 先闭环，后扩展

第一阶段只追求最小可用闭环，不追求功能齐全。

### 2. 参考 OpenClaw，但必须独立

可以参考 OpenClaw 的思路，但交付物必须是：

* 独立仓库
* 独立运行
* 不要求 OpenClaw 作为 runtime 宿主
* 不把 OpenClaw 直接硬绑进来

### 3. Telegram 是主入口

Telegram 不是附属接口，而是主要控制面。

### 4. 记忆必须服务连续任务执行

不是为了“有记忆”而做记忆，而是为了：

* 记住用户长期偏好
* 记住 OpenEmotion 项目背景
* 记住任务状态
* 让 `/resume` 真能接着跑

### 5. 配置层必须第一阶段一起做好

不能把模型、API、提示词、工具开关等硬编码在代码里。
第一阶段必须把 **配置层** 做成正式能力，否则后面会迅速失控。

### 6. 任务执行优先于聊天能力

这个系统首先要是“能做事的 agent”，不是“会说话的 bot”。

## 五、第一阶段必做模块

---

### A. Telegram Adapter

职责：

* 接收 Telegram 消息
* 发送回复
* 支持命令与自然语言入口
* 作为主控制界面

至少支持命令：

* `/new`
* `/status`
* `/tasks`
* `/resume`
* `/pause`
* `/retry`
* `/abort`
* `/report`
* `/memory`

要求：

* 能稳定收发消息
* 命令路由清晰
* 自然语言消息可进入任务创建流程

---

### B. Task Runtime

职责：

* 创建任务
* 保存任务目标
* 保存任务状态
* 管理 step-based 执行
* 支持暂停 / 恢复 / 重试 / 中止
* 输出当前摘要和下一步建议

建议状态：

* created
* planning
* running
* paused
* blocked
* completed
* failed
* aborted

要求：

* 任务状态真实可跟踪
* `/status`、`/resume`、`/report` 可直接使用
* 每次推进后都能保存状态

---

### C. Tool Bus

第一阶段至少实现：

* file tool
* shell tool
* python tool

要求：

* 工具统一注册
* 工具统一调用接口
* 返回结构化结果
* 标准错误处理
* 有执行日志
* 有基本安全边界（工作目录限制、超时、输出截断）

可以参考 OpenClaw 的工具抽象方式，但实现必须独立。

---

### D. Memory Core

第一阶段只做四类记忆：

#### 1) Profile Memory

用户长期偏好、默认规则、稳定背景

#### 2) Project Memory

OpenEmotion 项目背景、结构说明、长期知识

#### 3) Task Memory

任务目标、当前进度、失败点、下一步、关键证据
这是第一阶段最重要的记忆层

#### 4) Interaction Memory

近期关键交互摘要、关键决策摘要

要求：

* 能写入
* 能读取
* 新任务创建时能注入相关上下文
* `/resume` 时 task memory 必须参与恢复

---

### E. Event Log + Checkpoint

要求：

* 关键动作写入 event log
* 每轮任务推进后保存 checkpoint
* 程序重启后可恢复最近任务的基本状态

建议事件类型：

* telegram_message_received
* task_created
* task_planned
* tool_called
* tool_succeeded
* tool_failed
* memory_written
* task_resumed
* task_completed

---

### F. OpenEmotion Bridge（轻量预留）

第一阶段不要做深度整合，但必须预留清晰接口：

* interaction outcome input
* task result summary input
* appraisal input placeholder
* reflection input placeholder
* narrative update placeholder
* trust/relation placeholder

要求：

* Runtime 不要把高层认知逻辑写死
* 后续能平滑接 OpenEmotion

---

### G. Config Layer（第一阶段必做）

这是新增硬要求，必须与主功能一起交付。

配置层必须至少覆盖：

#### 1) App 基础配置

例如：

* app name
* environment
* log level
* data path
* workspace path
* checkpoint path

#### 2) Telegram 配置

例如：

* bot token
* allowed chat ids
* admin ids
* polling / webhook 模式
* message size limit

#### 3) LLM / API 配置

例如：

* provider 列表
* 默认 provider
* 默认 model
* 每个模型的参数
* timeout
* retry
* fallback 策略（第一阶段可做最小版）

#### 4) Prompt 配置

例如：

* 主 system prompt
* planner prompt
* executor prompt
* report prompt
* memory summary prompt
* resume prompt

#### 5) Tool 配置

例如：

* 启用的工具列表
* shell timeout
* 工作目录白名单
* 文件读写边界
* Python 执行限制

#### 6) Memory 配置

例如：

* 各类记忆存储策略
* 摘要长度
* 注入策略
* 最近任务窗口长度

#### 7) OpenEmotion Bridge 配置

例如：

* bridge 开关
* 预留字段
* 未来桥接参数入口

## 六、配置层的实现要求

### 敏感信息与结构化配置分离

必须采用分层配置：

* **`.env` / `.env.example`**：放敏感信息
  例如 API keys、Telegram token
* **`config/*.yaml`**：放结构化配置
  例如模型、参数、提示词、工具开关、路径配置

### 不允许的做法

* 不允许把模型名硬编码散落在代码里
* 不允许把 prompt 直接散落在各业务文件中
* 不允许把 provider / model / temperature / timeout 等写死在 runtime 逻辑里
* 不允许把 Telegram token、API key 等直接写进代码

### 配置层必须具备的能力

* 启动时加载配置
* 基础配置校验
* 默认值支持
* 模块化读取
* README 中说明如何修改模型、参数、提示词

* `system_main`

## 九、推荐技术方向

### 语言

优先建议 **Python**

原因：

* 更适合快速闭环
* Telegram / SQLite / JSONL / 文件处理简单
* 更利于后续接 OpenEmotion
* 更适合先做最小 runtime

### 存储

* **SQLite**：任务、记忆索引、状态表
* **JSONL**：event log
* **JSON / Markdown**：记忆和报告补充材料

### 运行方式

* 单进程优先
* 先跑通闭环
* 第一阶段不要上来就拆复杂服务

## 十一、开发任务分解

---

### Phase 1：项目骨架 + 配置层基础

1. 初始化独立仓库
2. 建立基础目录结构
3. 建立配置加载器 `config.py`
4. 建立 `.env.example`
5. 建立 `config/*.yaml`
6. 建立基础日志系统
7. 提供可运行入口 `main.py`

交付标准：

* 程序可启动
* 配置可读取
* 缺失关键配置时有清晰报错
* 默认值与校验机制可工作

---

### Phase 2：Telegram 接入

1. 接入 Telegram bot
2. 能接收文本消息
3. 能发送文本回复
4. 建立 command router
5. 识别 `/new`、`/status`、`/resume`、`/tasks`

交付标准：

* Telegram 中可正常与 bot 交互
* 基础命令可工作
* 自然语言消息可进入任务入口

---

### Phase 3：任务执行内核

1. 定义 task / task_step 数据模型
2. 定义任务状态机
3. 实现任务创建
4. 实现最小 planning
5. 实现一步执行 + 状态推进
6. 实现 `/report`

交付标准：

* 能创建任务
* 能看到任务状态变化
* 能输出“已完成 / 当前 / 下一步”摘要

---

### Phase 4：工具系统

1. 实现 tool registry
2. 实现 file tool
3. 实现 shell tool
4. 实现 python tool
5. 工具执行结果结构化返回
6. 工具调用写入事件日志

交付标准：

* agent 至少能完成简单目录检查、文件读取、受控命令执行
* 工具失败不会让 runtime 崩溃

---

### Phase 5：记忆系统

1. 实现 memory manager
2. 实现四类记忆
3. 任务执行后自动写入 task memory
4. `/resume` 自动读取 task memory
5. 新任务创建时注入 profile/project memory

交付标准：

* 任务连续性成立
* resume 不是重新开始
* 任务摘要可复用

---

### Phase 6：Checkpoint 与恢复

1. 实现 event log
2. 实现 checkpoint 持久化
3. 程序重启后恢复最近任务状态
4. `/status` 与 `/resume` 恢复后可用

交付标准：

* 最近任务基本状态不丢
* 至少可恢复 1 个进行中的任务

---

### Phase 7：LLM / Prompt 配置接入

1. 实现 llm client
2. 读取 llm.yaml 配置
3. 读取 prompts.yaml 配置
4. planner / executor / report 使用配置化 prompt
5. provider / model / 参数从配置读取，而不是硬编码

交付标准：

* 修改配置即可切换模型或参数
* 修改 prompt 文件即可影响 planner/executor/report 行为
* README 中说明如何改模型与 prompt

---

### Phase 8：OpenEmotion Bridge 预留

1. 实现 bridge 接口
2. 将 interaction outcome / task result summary 暴露给 bridge
3. 保持轻耦合

交付标准：

* 后续可平滑接入 OpenEmotion 高层能力

---

### Phase 9：闭环验证

至少完成 3 个端到端场景：

#### 场景 A：目录检查任务

“检查 OpenEmotion 项目目录结构，并告诉我下一步建议。”

#### 场景 B：文件分析任务

“读取 README 或 roadmap 文件并总结关键点。”

#### 场景 C：继续执行任务

“继续上一个任务” 或 `/resume`

交付标准：

* 三个场景都能跑通
* Telegram 中可真实使用
* 任务状态、记忆、checkpoint 一致
* 重启后至少一个任务可恢复

## 十三、验收标准

第一阶段通过标准只有一个核心判断：

**这个 agent 是否已经能在 Telegram 中真实接任务、利用记忆维持连续性、通过配置化模型与提示词推进任务、并把简单任务闭环跑完。**

满足以下即视为通过：

* Telegram 收发正常
* 新任务可创建
* 状态可查询
* 至少 2~3 个工具可调用
* Task memory 可保存与恢复
* 程序重启后可恢复基本状态
* `/resume` 能接着推进
* 模型 / 参数 / prompt 可通过配置修改
* 配置不散落在代码中
* 能输出阶段性任务报告
