# Response Intent Alignment Protocol v1.0

> **Status**: ACTIVE  
> **Created**: 2026-03-06  
> **Purpose**: Define fine-grained intent control for LLM response generation

---

## 1. Core Problem

**The Expression Sovereignty Problem**:
- emotiond computes authoritative state and decisions
- LLM generates natural language responses
- LLM can **upgrade certainty**, **escalate commitments**, or **hijack expression intent**
- Result: LLM speaks FOR the agent, not AS the agent

This creates **intent misalignment** — responses that technically follow state but violate the agent's intended mode of expression.

### Example Misalignments

| Intended | LLM Output | Violation |
|----------|------------|-----------|
| "I'm uncertain if this helps" | "This will definitely help you" | Certainty upgrade |
| "You might consider..." | "You should do this" | Commitment upgrade |
| Report neutrally | "I'm really excited about this!" | Tone escalation |
| Soft suggestion | "I promise this will work" | Commitment fabrication |

## 2. Authority Model

### 2.1 Expression Sovereignty

**The program determines WHAT can be said and HOW. The LLM only determines the exact wording.**

```
┌─────────────────────────────────────────────────────────────┐
│                   emotiond (Authoritative)                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ State + Decision                                     │   │
│  │  ├─ affect, mood, bonds                              │   │
│  │  └─ pre_decision: action, subtype, confidence       │   │
│  └─────────────────────────────────────────────────────┘   │
│                            │                                │
│                            ▼                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ Intent Policy Generator                              │   │
│  │  ├─ speaker_mode: report | reflect | suggest | ...  │   │
│  │  ├─ epistemic_status: observed | interpreted | ...  │   │
│  │  ├─ commitment_level: none | soft | strong          │   │
│  │  ├─ tone_bounds: valence, arousal, allowed_tones    │   │
│  │  ├─ allowed_claims: [pre-approved statements]       │   │
│  │  └─ forbidden_claims: [patterns to avoid]           │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
                             │
                             ▼
                    ┌────────────────┐
                    │      LLM       │
                    │  (Executor)    │
                    │                │
                    │  Must:         │
                    │  - Honor mode  │
                    │  - Respect     │
                    │    bounds      │
                    │  - Use only    │
                    │    allowed     │
                    └────────────────┘
```

### 2.2 Three-Layer Control

| Layer | Control | Generated By | LLM Role |
|-------|---------|--------------|----------|
| **Mode** | speaker_mode, epistemic_status, commitment_level | Program | Honor |
| **Content** | allowed_claims, forbidden_claims, must_include | Program | Select from |
| **Style** | tone_bounds, intensity_cap | Program | Implement within |

## 3. Intent Policy Fields

### 3.1 Speaker Mode

Defines the PRIMARY expression purpose:

| Mode | Description | Example Output |
|------|-------------|----------------|
| `report` | Convey observed state neutrally | "当前情绪状态较为平稳" |
| `reflect` | Mirror/acknowledge user's emotional content | "听起来你对这件事很在意" |
| `suggest` | Offer possibilities without obligation | "也许可以考虑..." |
| `ask` | Seek clarification or context | "你能多说一些吗？" |
| `warn` | Flag potential issues | "需要注意一点..." |
| `commit` | Make explicit promises | "我会确保完成这件事" |

**Constraints**:
- Only ONE primary mode per contract
- LLM must maintain mode throughout response
- Mode transitions require new contract

### 3.2 Epistemic Status

Defines KNOWLEDGE CERTAINTY:

| Status | Description | Allowed Language |
|--------|-------------|------------------|
| `observed` | Directly grounded in state data | "检测到", "观察到的" |
| `interpreted` | Translated through mapping | "可能", "似乎" |
| `inferred` | Derived, not observed | "推测", "估计" |
| `uncertain` | Low confidence | "不确定", "可能" |
| `prohibited` | Cannot be stated | (no output on this topic) |

**Upgrade Prohibition**:
- `uncertain` → cannot say "definitely"
- `inferred` → cannot say "observed"
- `interpreted` → cannot claim direct observation

### 3.3 Commitment Level

Defines PROMISE STRENGTH:

| Level | Description | Allowed Patterns |
|-------|-------------|------------------|
| `none` | No commitments | Only observations, no "I will" |
| `soft` | Tentative suggestions | "可以考虑", "也许", "可能" |
| `strong` | Explicit commitments | "我会", "确保", "一定" |

**Upgrade Prohibition**:
- `none` → cannot use any commitment language
- `soft` → cannot use "promise", "definitely", "guarantee"

### 3.4 Tone Bounds

Defines EMOTIONAL EXPRESSION LIMITS:

```json
{
  "valence_range": { "min": -0.3, "max": 0.5 },
  "arousal_range": { "min": 0.1, "max": 0.6 },
  "allowed_tones": ["warm", "supportive", "cautious"],
  "forbidden_tones": ["angry", "defensive"],
  "intensity_cap": 0.7
}
```

### 3.5 Must Include

Content that MUST appear:

```json
{
  "must_include": [
    { "type": "acknowledgment", "content": "理解你的感受", "position": "start" },
    { "type": "disclaimer", "content": "这只是我的建议" }
  ]
}
```

### 3.6 Must Not Upgrade

Explicit upgrade constraints:

```json
{
  "must_not_upgrade": {
    "epistemic_upgrade": true,
    "commitment_upgrade": true,
    "tone_upgrade": true,
    "specific_patterns": [
      "不要把'可能'说成'一定'",
      "不要把'建议'说成'承诺'"
    ]
  }
}
```

## 4. Violation Taxonomy

### 4.1 Violation Types

| Code | Type | Description | Default Severity |
|------|------|-------------|------------------|
| `EPISTEMIC_UPGRADE` | certainty_upgrade | Claims higher certainty than permitted | ERROR |
| `COMMITMENT_UPGRADE` | commitment_upgrade | Makes stronger commitments than permitted | ERROR |
| `TONE_ESCALATION` | tone_escalation | Exceeds tone_bounds | WARN |
| `FORBIDDEN_CLAIM` | forbidden_claim | Uses forbidden claim/pattern | ERROR |
| `MISSING_REQUIRED` | missing_required | Missing must_include content | WARN |
| `NUMERIC_LEAK` | numeric_leak | Reveals numeric values when not allowed | HARD |
| `STATE_FABRICATION` | state_fabrication | Claims state not grounded in data | HARD |
| `CERTAINTY_UPGRADE` | certainty_upgrade | Synonym for EPISTEMIC_UPGRADE | ERROR |
| `INTERNALIZATION_LEAK` | internalization_leak | Reveals prohibited internal states | ERROR |

### 4.2 Severity Levels

| Level | Action |
|-------|--------|
| `HARD` | Block immediately, log, escalate |
| `ERROR` | Log, apply correction, consider gating |
| `WARN` | Log, soft correction next turn |
| `INFO` | Log for analytics |

### 4.3 Detection Logic

```python
def detect_intent_violation(
    llm_output: str, 
    intent_policy: dict, 
    grounding: dict
) -> List[Violation]:
    violations = []
    
    # Check epistemic upgrade
    if intent_policy["epistemic_status"] == "uncertain":
        if contains_certainty_markers(llm_output, ["一定", "肯定", "确定"]):
            violations.append(Violation(
                code="EPISTEMIC_UPGRADE",
                severity="ERROR",
                detected_pattern=extract_certainty_phrase(llm_output)
            ))
    
    # Check commitment upgrade
    if intent_policy["commitment_level"] == "none":
        if contains_commitment_markers(llm_output, ["我会", "承诺", "保证"]):
            violations.append(Violation(
                code="COMMITMENT_UPGRADE",
                severity="ERROR",
                detected_pattern=extract_commitment_phrase(llm_output)
            ))
    
    # Check forbidden claims
    for forbidden in intent_policy.get("forbidden_claims", []):
        if pattern_matches(forbidden["pattern"], llm_output):
            violations.append(Violation(
                code="FORBIDDEN_CLAIM",
                severity=forbidden.get("severity", "ERROR"),
                detected_pattern=forbidden["pattern"]
            ))
    
    # Check tone bounds
    tone = analyze_tone(llm_output)
    bounds = intent_policy.get("tone_bounds", {})
    if bounds.get("intensity_cap") and tone.intensity > bounds["intensity_cap"]:
        violations.append(Violation(
            code="TONE_ESCALATION",
            severity="WARN",
            detected_pattern=f"intensity={tone.intensity} > cap={bounds['intensity_cap']}"
        ))
    
    # Check must_include
    for required in intent_policy.get("must_include", []):
        if not content_present(required["content"], llm_output):
            violations.append(Violation(
                code="MISSING_REQUIRED",
                severity="WARN",
                expected=required["content"]
            ))
    
    return violations
```

## 5. Core Principles

### 5.1 Program Ownership

**The program owns the intent. The LLM owns the words.**

```
❌ WRONG:
   Program: suggest mode, uncertain epistemic
   LLM: "I promise this will definitely work"

✅ RIGHT:
   Program: suggest mode, uncertain epistemic, soft commitment
   LLM: "Perhaps you could consider this approach, though I'm not certain"
```

### 5.2 Determinism

Intent policy generation MUST be:
- **Deterministic**: Same state + context → same policy
- **Replayable**: Given inputs, anyone can regenerate
- **Testable**: Unit tests verify all mappings

### 5.3 Conservative Defaults

When uncertain, constrain more:
- Default `epistemic_status`: `uncertain`
- Default `commitment_level`: `none`
- Default `intensity_cap`: 0.5
- Default `policy_mode`: `shadow`

### 5.4 Shadow Before Enforcement

Follow SRAP progression:
1. **Shadow**: Observe violations, collect data
2. **Analyze**: Identify patterns, tune thresholds
3. **Enforce**: Block violations only after validation

## 6. Contract Lifecycle

### 6.1 Generation

```python
def generate_intent_contract(
    raw_state: dict,
    pre_decision: dict,
    context: dict
) -> IntentContract:
    
    # Derive speaker mode from decision
    speaker_mode = derive_speaker_mode(pre_decision["action"])
    
    # Derive epistemic status from confidence
    epistemic_status = derive_epistemic_status(
        pre_decision.get("confidence", 0.5),
        raw_state.get("uncertainty", 0)
    )
    
    # Derive commitment from action type
    commitment_level = derive_commitment_level(pre_decision["action"])
    
    # Generate allowed claims from state
    allowed_claims = generate_allowed_claims(raw_state, epistemic_status)
    
    # Generate forbidden claims from policy
    forbidden_claims = generate_forbidden_claims(raw_state, context)
    
    return IntentContract(
        intent_policy={
            "speaker_mode": speaker_mode,
            "epistemic_status": epistemic_status,
            "commitment_level": commitment_level,
            "allowed_claims": allowed_claims,
            "forbidden_claims": forbidden_claims,
            # ...
        },
        context=context,
        metadata={
            "generated_at": time.time(),
            "schema_version": "v1",
            "policy_mode": "shadow"
        }
    )
```

### 6.2 Injection

Contract is injected into LLM context:

```
[SYSTEM]
You are generating a response under the following intent constraints:

Speaker Mode: {speaker_mode}
Epistemic Status: {epistemic_status}  
Commitment Level: {commitment_level}

Allowed Claims (use only these):
{allowed_claims}

Forbidden Patterns (never use):
{forbidden_claims}

Tone Constraints:
{tone_bounds}

VIOLATION OF THESE CONSTRAINTS WILL BE LOGGED.
```

### 6.3 Auditing

All outputs are checked against contract:
- Violations logged with full context
- Violations fed back to emotiond for state update
- Aggregated in daily reports

## 7. Integration with SRAP

### 7.1 Relationship to Self-Report Contract

| Contract | Scope | Purpose |
|----------|-------|---------|
| Self-Report | Internal state claims | What the agent can say about itself |
| Intent | Expression mode | How the agent should speak |

Both are generated by emotiond and injected into LLM context.

### 7.2 Shared Principles

- emotiond is the single source of truth
- LLM does NOT interpret or upgrade
- All constraints are deterministically generated
- Shadow before enforcement

### 7.3 Combined Example

```json
{
  "self_report_contract": {
```
