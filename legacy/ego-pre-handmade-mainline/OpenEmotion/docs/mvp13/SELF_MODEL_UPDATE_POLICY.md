# Self-Model Update Policy

## 1. Update Philosophy

Self-model updates must be conservative, evidence-linked,
and reversible where possible.

The system must not rewrite its identity based on a single
surface-level output.

## 2. Valid Update Triggers

Updates may be triggered by:
- repeated behavioral evidence
- stable developmental cycle patterns
- verified long-horizon consistency
- explicit governance-approved revision
- contradiction resolution

## 3. Invalid Update Triggers

Updates must NOT be triggered by:
- one-off textual self-description
- unverified hallucinated self-claims
- isolated user wording imitation
- ungrounded introspective guesswork

## 4. Update Modes

Suggested modes:
- append_observation
- revise_tendency
- resolve_tension
- promote_to_stable_trait
- mark_for_review
- rollback_revision

## 5. Confidence Thresholds

Field changes should depend on confidence class:
- low confidence → observation only
- medium confidence → tentative tendency
- high confidence → promotable with audit
- critical identity field → requires special review gate

## 6. Audit Requirements

Each update must write:
- before snapshot
- after snapshot
- diff
- trigger source
- supporting evidence
- confidence level
