任务：收敛 OpenEmotion × OpenClaw 集成拓扑（补齐 enforcer + 统一入口 + 脚本/文档一致性）
背景与问题（已定位）

当前 feature-emotiond-mvp 分支存在集成拓扑断裂与口径不一致：

文档/测试说明中引用了 emotiond-enforcer（pre-send 强制层），但仓库里缺失对应目录/配置，导致“不可绕过”可能只是纸面成立。

安装脚本仅安装 emotiond-bridge，未安装 enforcer，导致新机器按脚本装完也不具备强制链路。

启动入口存在多套说法（Makefile / README / CI），容易出现“本地跑的是 A，CI 跑的是 B”的隐性偏差。

token 文件/环境变量读取路径存在分裂，容易出现“看似跑通，实际权限层没生效”。

target_id / counterparty_id 语义在文档与 API 使用上容易混用，存在隔离/关系账本串台风险。

目标：让仓库成为单一事实源，任何人 clone 后通过脚本即可装齐 hooks，并且入口/文档/CI/回放全部一致，强制链路真实存在。

实现 “可复现闭环”：

git clone + scripts/install_openclaw_integration.sh 一次执行后

emotiond-bridge 和 emotiond-enforcer 都被安装到 OpenClaw hooks 目录

token 读取策略统一：脚本写入的位置、服务端读取位置、文档示例一致

enforcer 链路真实生效：pre-send 时能调用 decision 接口并执行 withdraw/boundary/attack（至少可在本地用一条模拟消息验证）

全量测试通过（现有 CI workflow 绿；本地 pytest 绿）

enforcer 最小行为要求（MVP）

读取消息上下文（来自 bridge 的存储/上下文文件/或 hook chain 的 shared state）

调用 emotiond 的 decision/plan API（你现在已有接口即可）

根据返回的 decision 执行：

withdraw：阻止发送 + 记录审计日志

boundary：替换/裁剪输出（可先实现为阻止发送并提示）

attack：同 withdraw（先不做攻击式输出，防误伤；后续再扩展）

无论允许/拒绝，都写一条 hook 侧审计日志，方便排错

如果你坚持 enforcer 不进入仓库：那就必须把 integrations 里的文档/TESTING 全部改为“部署产物不在仓库”，并提供脚本生成它。但这条会让复现成本上升，不建议。

风险提示（提前规避）

enforcer 的“替换输出/裁剪”策略容易引入不可控副作用：MVP 阶段建议先实现 block + audit，把边界行为做成“拒绝发送并提示”，后续再逐步引入可控的重写。

token 多来源并存最容易让安全层形同虚设：务必只留一个权威路径 + env var 优先级。
