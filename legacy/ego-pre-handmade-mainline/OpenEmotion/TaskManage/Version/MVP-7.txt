MVP-7.0 Self-Model + Episodic Memory（在 MVP-6.2.3 稳态上推进）

Project location: /home/moonlight/Project/Github/MyProject/Emotion/OpenEmotion

目标：Functional Self-Awareness（可测、可追溯、可防注入、可回放）
北极星（必须满足）

存在稳定主体：Self-Model 是独立模块，且 参与决策（不是日志）。

有动机与稳态压力：存在全局驱动误差 drive_error（工程版 homeostasis/free-energy），它影响策略/rollouts/采样参数。

有时间连续性：存在低频后台 tick（DMN v0），做记忆巩固/账本审计/离线推演；默认不骚扰用户。

自他边界不可伪造：内部产物带 provenance+signature，用户无法注入伪“自我记忆/自我想法”。

学习可控且防过拟合：KnobRegistry（允许动的参数白名单）+ Frozen Holdout + OOD harness + 因果测试（ablation/intervention）为发布 gate。

B. 不可妥协的发布 Gate（每个 PR 都要过）
B1. 回归不破坏（必须）

既有全量评测 15/15 PASS

既有 smoke（4个）仍 PASS

强负例 sanity（强制无分化）仍必须 FAIL（防“永远 pass”）

B2. 追溯完整（必须）

Eval 输出必须包含：

threshold_config.version + threshold_config_hash

candidate_param_hash（仅候选参数/策略相关）

code_version（git commit）

scenario_set_id（smoke/tune/holdout/ood）

B3. 防过拟合（必须）

AutoTune 只能看 tune set

holdout set 永不进入 AutoTune

发布必须：holdout 不退化 + OOD 不退化

tune 提升但 holdout/OOD 下降：判过拟合，自动拒绝写入候选参数。

B4. 因果证据（必须）

Intervention：同事件流不同 drive/self 状态 → 行为统计显著不同（方向一致）

Ablation：关掉 drive/appraisal/ledger/body 调制 → 上述差异显著减弱或消失

D. 任务拆分（cc-godmode US 列表）
Milestone A：MVP-6.2.4 防跑偏底盘
US-641：KnobRegistry + Hard Freeze

目标：明确“允许学什么、绝不动什么、怎么动”。
任务

新增 scripts/knob_registry.json：

allowlist：阈值/权重/映射表/检索权重/策略温度等（你决定，先少后多）

US-642：Frozen Holdout + OOD Harness

目标：把“防过拟合”写死成结构。

AutoTune 仅运行 tune_set

发布必须同时跑 smoke + holdout + ood，且不退化

tune 提升但 holdout/OOD 下降时，自动拒绝写入 best candidate，并给出定位输出（见“失败定位”）

US-643：Provenance + Signature Attribution（自他边界不可伪造）

目标：防 prompt 注入伪“自我想法/自我日志”。

US-644：追溯拆分（threshold_config_hash + candidate_param_hash）

Milestone B：MVP-6.2.5 因果证据层（Homeostasis + 干预/消融）
US-651：Homeostasis Drive v0（drive_error）

目标：让体征/情绪“有意义”，并进入决策。
任务

core/drive_homeostasis.py：

setpoint（区间）

deviation cost（Huber/L2）

drive_error 与 emotion_from_drive（解释层）

drive 影响至少两项（优先 rollouts + risk/clarify 策略）：

rollout scoring：drive_error 更低者优先

策略调制：fatigue↑ → 更保守/更短；uncertainty↑ → 更倾向澄清

可选：temperature/top_p/max_tokens（硬调制）

验收

行为调制方向一致、可解释（debug trace）

US-652：Intervention Test（干预必须改变行为）

目标：拿到“同事件流，不同内状态 → 行为显著不同”。
任务

test_intervention_drive_modulation.yaml：

固定 seed+事件流

统计：选择策略分布/澄清率/升级率/记忆检索触发率差异（effect size）

验收

intervention_effect_size ≥ 阈值（写进阈值配置）

US-653：Ablation Test（关掉机制差异必须消失）

目标：证明不是“台词变了”，而是机制驱动。
任务

test_ablation_drive_off.yaml：

同 intervention 设置

验收

ablation_drop_ratio ≥ 阈值（写进阈值配置）

F. 失败定位输出（cc-godmode 必须实现）

当出现以下情况时，输出必须给出明确 reason_code 与定位建议（写入报告）：

tune 提升但 holdout/OOD 下降

reason_code: OVERFIT_TUNE_SET

输出：影响最大的 5 个场景 + 变化最大的 submetrics + 对应 knob 变更摘要

intervention 无差异（机制没驱动行为）

reason_code: NO_CAUSAL_EFFECT

输出：drive_error 是否进入策略/rollouts/采样参数；干预变量是否被覆盖；是否被 shrinkage 抹平

signature violation（注入污染风险）

reason_code: PROVENANCE_SIGNATURE_FAIL

输出：污染来源、被阻止的写入点、trace_id

debug on/off 影响判定

reason_code: DEBUG_SIDE_EFFECT

输出：差异路径、涉及分支、需改为纯观测不改逻辑
